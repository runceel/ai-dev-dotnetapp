# Visual Studio 2026 GitHub Copilot Agent Mode ハンズオン

> 対象: EventRegistration リポジトリを題材にした 30 分ハンズオン  
> 想定環境: Visual Studio 2026 + GitHub Copilot Agent mode  
> 主題: 既存 .NET アプリケーションの調査、計画、実装、検証を Agent mode で進める

---

## 概要

このドキュメントは、Visual Studio 2026 の GitHub Copilot Agent mode を使い、初見の既存 .NET ソリューションに対して機能追加を行う 30 分ハンズオンの進行案を定義します。

題材はイベント参加登録システムです。参加者は、Visual Studio 2026 に開かれた既存ソリューションを Copilot Agent に調査させ、実装計画を確認したうえで、イベント一覧画面に参加状況を追加します。

### 主要なポイント

| 項目 | 内容 |
|------|------|
| 主な体験 | Agent mode による既存コード調査、計画作成、複数ファイル編集、テスト確認 |
| 使用しない主動線 | Visual Studio Code、GitHub Copilot CLI、Inline Chat 中心の進行 |
| 実装テーマ | イベント一覧カードに参加確定者数、残り枠数、満席表示、キャンセル待ち有無を追加 |
| 重点 | Plan mode が使えない環境でも、Agent mode への指示で「調査だけ」「計画だけ」「実装」「検証」を分ける |
| 成果物 | 動作する UI 変更、bUnit テスト追加または更新、レビュー向け変更説明 |

> 注: Visual Studio 2026 の Plan mode が利用できる環境では、Agent mode と Plan mode を切り替えることで、本ハンズオンの「計画だけ作らせる」手順をより明示的に実行できます。ただし、現時点では Plan mode がプレビュー版で利用できない環境もあるため、本資料では Agent mode に対して「まだコードは変更しないでください」と明示する運用を主手順とします。

---

## 1. ハンズオンの狙い

このハンズオンでは、Copilot を単なるコード生成ツールではなく、既存プロジェクトに入るときの共同作業者として使う流れを扱います。

参加者が持ち帰るべき型は次の通りです。

1. 既存ソリューションを Agent に調査させる
2. 変更前に実装計画を出させる
3. 人間が計画をレビューし、スコープを絞る
4. Agent に実装とテスト更新を任せる
5. ビルド・テスト・動作確認まで依頼する
6. 変更内容をレビュー向けに説明させる

特に重要なのは、いきなり実装させないことです。Agent mode であっても、指示を段階化すれば、Plan mode がない環境でも安全な進め方を再現できます。

## 2. 前提条件

### 2.1 参加者環境

| 項目 | 内容 |
|------|------|
| IDE | Visual Studio 2026 |
| Copilot | GitHub Copilot Agent mode が利用可能です |
| SDK | .NET 10 SDK |
| ソリューション | `EventRegistration.slnx` を Visual Studio 2026 で開いていること |
| 推奨確認手段 | Visual Studio の Solution Explorer、GitHub Copilot Chat、Test Explorer、Aspire AppHost 起動 |

### 2.2 講師側の準備

- `main` ブランチの状態から開始できるようにします
- Visual Studio 2026 で `EventRegistration.slnx` を開けることを確認します
- `EventRegistration.AppHost` をスタートアップ対象として起動できることを確認します
- デモデータが表示される Development 設定で起動します
- Test Explorer で `EventRegistration.Web.Tests` のテストを実行できることを確認します

## 3. 30 分進行案

| 時間 | セクション | 内容 |
|------|------------|------|
| 0-3 分 | 導入 | 題材、ゴール、Agent mode 主体の進め方を説明します |
| 3-8 分 | 調査 | Agent にソリューション構成、画面、関連テストを調査させます |
| 8-13 分 | 計画 | Agent に実装計画だけを作らせ、人間がレビューします |
| 13-22 分 | 実装 | 計画に沿ってイベント一覧の参加状況表示とテストを実装させます |
| 22-27 分 | 検証 | ビルド、テスト、Aspire 起動で確認します |
| 27-30 分 | まとめ | Agent にレビュー向け説明を作らせ、実務での使い方に接続します |

時間が不足した場合は、Aspire 起動による画面確認を講師デモに切り替え、参加者はテスト確認までをゴールにします。

## 4. 題材にする変更

### 4.1 変更内容

イベント一覧画面のカードに、各イベントの参加状況を表示します。

表示項目は次の通りです。

| 表示項目 | 説明 |
|----------|------|
| 参加確定者数 | `RegistrationStatus.Confirmed` の件数 |
| 残り枠数 | イベント定員から参加確定者数を引いた数。0 未満にはしない |
| 満席表示 | 残り枠数が 0 の場合に表示します |
| キャンセル待ち表示 | `RegistrationStatus.WaitListed` が 1 件以上ある場合に表示します |

### 4.2 この題材が適している理由

- 既存 UI の変更として見た目の成果が分かりやすいです
- Events と Registrations の関係を調べる必要があり、既存コード調査の意味が出ます
- Modular Monolith / Clean Architecture の依存方向を崩さない設計判断を扱えます
- bUnit テストが既にあり、既存テストスタイルに合わせた追加を体験できます
- 30 分でも完走しやすい範囲に収まります

## 5. Agent mode へのプロンプト例

### 5.1 調査だけを依頼する

```text
このソリューションを調査してください。

知りたいこと:
- 主要なプロジェクト構成
- アーキテクチャの方針
- Events / Registrations モジュールの役割
- イベント一覧画面の実装場所
- 参加登録情報を取得している UseCase や Repository
- 関連するテストの場所

まだコードは変更しないでください。
調査結果だけを簡潔にまとめてください。
```

講師は、Agent がどのファイルを読んだか、どのように関連箇所を見つけたかを見せます。

### 5.2 実装計画だけを依頼する

```text
イベント一覧画面に、各イベントの参加状況を表示したいです。

表示したい内容:
- 参加確定者数
- 残り枠数
- 満席かどうか
- キャンセル待ちがあるかどうか

既存の Modular Monolith / Clean Architecture / MudBlazor / bUnit の方針に従ってください。

まず実装計画を立ててください。
まだコードは変更しないでください。

計画には次を含めてください:
- 変更対象ファイル
- 設計方針
- テスト方針
- 想定リスク
```

この段階では、実装の正しさよりも、変更範囲が適切かを確認します。

### 5.3 計画を小さく調整する

Agent が大きな抽象化、新規プロジェクト、過剰なリファクタリングを提案した場合は、次のようにスコープを絞ります。

```text
計画をもう少し小さくしてください。

今回は30分ハンズオンなので、イベント一覧画面と既存テストの更新を中心にしてください。
新しいプロジェクトや大きな抽象化は追加しない方針で再計画してください。
既存の UseCase と UI パターンを優先してください。

まだコードは変更しないでください。
```

### 5.4 実装を依頼する

```text
この計画で実装してください。

既存のコードスタイルに合わせてください。
必要な bUnit テストも追加または更新してください。
実装後に、変更内容を簡潔に報告してください。
```

実装中は、Agent の提案する差分を確認し、必要に応じて承認します。講師は、複数ファイル変更を Agent mode に任せる体験を強調します。

### 5.5 検証を依頼する

```text
ビルドとテストを実行して確認してください。
失敗した場合は原因を調査して修正してください。

確認結果を簡潔に報告してください。
```

Visual Studio 2026 では、Test Explorer とビルド出力も併用して確認します。

### 5.6 レビュー向け説明を依頼する

```text
今回の変更をレビュー担当者向けにまとめてください。

含めてほしい内容:
- 変更したファイル
- 設計上の判断
- 追加または更新したテスト
- 確認結果
- 残っている注意点

簡潔に説明してください。
```

この出力は、Pull Request の説明文やチーム共有の下書きとして利用できます。

## 6. 講師用ガイド

### 6.1 見せるべきポイント

| ポイント | 説明 |
|----------|------|
| Agent に任せる前に調査させる | 初見コードへの入り方を示します |
| 計画を人間がレビューする | Agent の出力をそのまま受け入れず、スコープを制御します |
| 既存パターンを優先させる | リポジトリの設計に合わせる重要性を示します |
| テストまで依頼する | コード生成だけで終わらせない流れを示します |
| 最後に説明させる | レビューや引き継ぎにも Copilot を使えることを示します |

### 6.2 期待される変更対象

実際の Agent の判断により差分は変わり得ますが、主に次のファイルが候補になります。

| ファイル | 目的 |
|----------|------|
| `src/EventRegistration.Web/Components/Pages/Events/EventList.razor` | イベント一覧カードに参加状況を表示します |
| `src/tests/EventRegistration.Web.Tests/Components/Pages/Events/EventListTests.cs` | 表示内容の bUnit テストを追加または更新します |

必要に応じて、`GetRegistrationsByEventUseCase` の注入やテスト用モックの追加が発生する可能性があります。

### 6.3 注意点

- 新しいモジュールや新しい Infrastructure を作る必要はありません
- Events モジュールから Registrations モジュールへ直接参照を追加する必要はありません
- Web 層で既存 UseCase を組み合わせる程度に留めると、ハンズオンとして扱いやすいです
- 大きな設計変更を Agent が提案した場合は、スコープを小さくするように指示します
- 参加者が Visual Studio Code や GitHub Copilot CLI の操作に寄らないよう、Visual Studio 2026 の UI を主動線にします

## 7. うまくいかなかった場合の代替ルート

### 7.1 時間が足りない場合

イベント一覧への表示項目を減らします。

最小構成は次の 2 つで十分です。

- 参加確定者数
- 残り枠数

満席表示とキャンセル待ち表示は、講師デモまたは宿題に回します。

### 7.2 Agent の変更が大きくなりすぎた場合

次のプロンプトで抑制します。

```text
変更範囲を小さくしてください。
今回は EventList.razor と EventListTests.cs を中心にしてください。
新しい抽象化や大きなリファクタリングは行わず、既存 UseCase を使って実装してください。
```

### 7.3 テストがすぐに通らない場合

講師側で次を確認します。

- `GetRegistrationsByEventUseCase` に必要な依存がテスト DI に登録されているか
- `IRegistrationRepository` のモックが追加されているか
- 非同期描画に対して `WaitForAssertion` を使っているか
- MudBlazor 用の `AddMudServices()` と JSInterop 設定が既存テストに合わせられているか

## 8. まとめメッセージ例

最後に参加者へ伝えるまとめは、次のようにします。

```text
Agent mode は、いきなり全てを任せるためのものではありません。
調査、計画、実装、検証を分けて依頼すると、Plan mode が使えない環境でも安全に既存プロジェクトへ変更を入れられます。

特に既存アプリでは、コードを書く前に「どこを触るべきか」「既存パターンは何か」「テストはどこにあるか」を Agent に調べさせることが重要です。
```
